# TUI GAME — Agent Guide

适配于终端的游戏引擎，基于 Rust + Lua。

## 项目概述
- **项目用途**：终端游戏引擎，支持 Lua 脚本驱动游戏逻辑
- **技术栈**：Rust + Lua (mlua)
- **仓库结构**：src/ 为核心源码目录，采用阶段化生命周期管理

## 项目结构
**开发目录**
```
src/host_engine/
├── core/          # 核心文件：时钟、帧管理、世界运行时、退出状态、panic钩子、引擎初始化、宿主状态机（宿主运行核心包含在内）
├── ui/            # 宿主 UI 界面：终端渲染、布局、事件处理
├── service/       # 所有服务：日志、配置、资源管理、Lua 绑定等
├── boot/          # 启动阶段：初始化顺序、依赖注入、配置加载
├── runtime/       # 运行阶段：主循环、帧更新、Lua 调度
└── shutdown/      # 结束阶段：资源释放、状态持久化、优雅退出
```

**上线目录**
```
[root]
├── tg          # 编译好的字节码文件
├── assets/     # 官方资源目录
├── data/       # 总数据目录
└── scripts/    # 官方脚本目录
```

## 快速命令
```bash
# 构建
cargo build

# 运行
cargo run

# 测试
cargo test

# 格式化
cargo fmt

# Lint
cargo clippy
```

## 关键约定
- **阶段分离**：boot → runtime → shutdown 三个阶段职责明确，禁止跨阶段耦合
- **Lua 绑定**：目前 Lua 服务只有占位，宿主稳定后实现
- **错误处理**：使用 service 里的 log 服务，所有的错误应当优先拦截并进入 log 服务

## 行为约束

1. **优先复用现有实现**  
  编码时优先使用现有的库、数据结构、算法及通用 API。仅当现有实现**明确无法支撑**当前客制化需求时，允许参考学习后进行改造。

2. **以当前需求为边界，禁止过度设计**  
  所有实现必须保证当前功能可用即可。除非某功能被明确标记为核心功能（放入 `core/`），否则不得为未来可能性提前扩展。若对"是否属于核心功能"存疑，按第 3 条执行。

3. **增量实现，逐块验证**  
  每个业务功能应优先完成当前模块的实现与测试，确认通过后再进入下一部分。避免多模块并行开发导致误差堆积，增加定位成本。

4. **遇到不确定必须停下并提问**  
  执行任务或制定计划时，若遇到任何不确定的问题，必须**立即暂停**并提出问题。提问时需附上：  
  - 问题现象描述  
  - 当前上下文（为什么会出现该问题）  
  - 你倾向的解决方案及理由（若有）

5. **代码精简原则**  
  提交前检查重复分支、重复数据转换、无意义包装和可以合并的相邻状态；精简不得牺牲命名、错误上下文和可测试性。

6. **可读性优先于过度优化**  
  代码压缩应始终以**人能看懂**为优先。禁止为追求代码量少而使用过长的链式调用、含义不明的临时变量名、过度复杂的函数间解耦，以及任何使代码难以理解的"炫技"写法。

7. **适度解耦，按调用次数判断**  
  当逻辑被复用，或单个函数同时承担多个可独立描述、测试的职责时进行拆分；避免仅为了缩短文件而创建无意义包装函数。

8. **忽略极低概率的边界情况**  
  处理边界情况前，先估算其发生概率。若某边界情况的发生概率**低到几乎不可能**，则视为无需处理。过度对齐边界只会导致代码复杂度无谓上升。
  安全边界、用户数据完整性、终端恢复和跨平台资源释放不以发生概率为由忽略。

9. **所有路径以上线目录为基准**
  开发环境仅作为测试目录使用，所有路径引用、资源加载、模块导入均以项目最终上线目录结构为基准进行编写，避免测试环境与线上环境路径不一致导致的运行时错误。
