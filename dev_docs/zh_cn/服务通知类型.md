## 查询

>现在是什么状态？

特点：
- 持续存在
- 多次查询结果可能相同
- 关心当前值而非变化
- 查询频率可能很高

## 事件队列

> 刚刚发生了什么？

特点：
- 瞬时、只发生一次
- 有先后顺序
- 跟随真实时间而非游戏帧
- 可能需要记录和回访

## 同步与异步

> 都要回答立即需要的结果

1. 如果是直接回答-同步：`直接返回值即可`
2. 如果是延迟回答-异步：`返回任务ID，不阻塞主线程，并在后续传入事件队列`

## 多线程

依照异步，不阻塞主线程进行任务，哪些适合：

适合：

- 网络请求；
- 文件读写；
- 图片解码和转换；
- 压缩与解压；
- 包扫描；
- 导出；
- 大型资源解析；
- 耗时路径搜索；
- 代码解析与高亮。

不适合：

- Lua 游戏逻辑；
- 画布修改；
- UI 对象池操作；
- 游戏世界修改；
- 输入事件分发；
- 状态机切换；
- 碰撞结果应用；
- 生命周期 API 调用。

# 回调和事件的本质区别

|特征|事件|回调|
|---|---|---|
|表达形式|数据|函数|
|耦合程度|较低|较高|
|是否容易记录|是|否|
|是否容易回放|是|否|
|是否可以广播|是|通常否|
|是否适合异步请求结果|是|是|
|Lua 热重载兼容性|较好|较难|
|存档恢复|可保存事件标识|无法保存函数闭包|
|使用便利性|稍繁琐|很方便|

对 TUI Game，我建议：

> **底层统一使用事件；Lua SDK 可以在事件之上提供回调语法糖。**

例如：

```
ctx.time.after(1000, function()
  spawn_enemy()
end)
```

底层实际保存：

```
TimerId → Lua RegistryKey
```

定时器到期时产生：

```
GameEvent::TimerFinished { timer_id }
```

主线程根据 `TimerId` 找到 Lua 回调并调用。

这样事件仍然是系统的真实通信机制，回调只是更方便的编程接口。
