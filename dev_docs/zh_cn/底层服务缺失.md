## 缺少正式的游戏实例

当前有 `RuntimeWorld`、`GameService` 和 Lua 服务，但还没有一个真正拥有完整生命周期的 `GameInstance`。

建议未来形成类似：

```
struct GameInstance {
    package: PackageInfo,
    lua: LuaRuntime,
    world: GameWorld,
    objects: GameObjectPool,
    resources: GameResources,
    permissions: CapabilitySet,
    state: GameInstanceState,
}
```

它应当明确拥有：

- 当前运行的包；
- 独立 Lua 环境；
- 游戏对象；
- 定时器和随机数；
- 输入动作映射；
- 资源句柄；
- 存档状态；
- 权限；
- 错误状态。

目前 `EngineServices` 是整个程序的服务集合，但还缺少“一个游戏运行期间独占哪些状态”的所有权边界。

这是最底层、最优先补齐的东西。

---

## 缺少正式的游戏生命周期协议

至少需要定义：

```
load
initialize
start
handle_event
fixed_update
update
render
pause
resume
save
stop
unload
```

不一定全部直接暴露给 Lua。对于当前需求，可以先简化为：

```
init(context)
event(state, event)
update(state, dt)
render(state)
save(state)
exit(state)
```

关键不在函数名字，而在宿主需要明确：

- 每个函数什么时候调用；
- 调用失败后怎么办；
- 返回值是否合法；
- 一帧内调用顺序是什么；
- 游戏暂停时哪些服务继续；
- 切换覆盖层时游戏是否更新；
- 窗口失焦是否暂停；
- 脚本超时如何中止；
- 游戏退出后如何释放资源。

---

## 缺少固定更新与可变更新的区分

当前有帧调度和 `delta_time`，但通用游戏运行时通常还要区分：

- 可变更新（variable update）：跟随实际帧率；
- 固定更新（fixed update）：以固定时间步长推进玩法；
- 渲染：根据最新状态输出画面。

例如：

```
输入采集
  ↓
0~N 次 fixed_update(16.67ms)
  ↓
update(real_dt)
  ↓
render(interpolation)
```

终端游戏同样需要它：

- 贪吃蛇移动速度不应随终端刷新率变化；
- 俄罗斯方块下降周期需要稳定；
- 碰撞和计时逻辑需要确定性；
- 帧率从 60 降到 30 时不能让游戏速度减半。

你已经有时间、定时器和帧调度基础，但还没有把它们统一成游戏调度阶段。

## 缺少统一的游戏对象模型

通用引擎一般使用：

- 场景树；
- 实体组件系统（ECS）；
- 游戏对象与组件；
- 或某种混合模型。

Godot 使用节点和场景树，Bevy 使用数据导向的实体组件系统；两种方案都在解决“游戏由什么对象构成、对象如何组合、谁拥有谁”。[Godot Node](https://docs.godotengine.org/en/stable/classes/class_node.html)、[Bevy ECS 导航](https://bevy.org/learn/quick-start/getting-started/)

TUI Game 暂时不需要完整 ECS。对于俄罗斯方块、数独、2048、扫雷等游戏，完整 ECS 很可能过度设计。

更适合的初版是：

```
struct GameWorld {
    next_id: GameObjectId,
    objects: HashMap<GameObjectId, GameObject>,
}
```

对象可以拥有少量可选组件：

```
struct GameObject {
    transform: Option<CellTransform>,
    visual: Option<CellVisual>,
    collider: Option<GridCollider>,
    animation: Option<CellAnimation>,
    tags: HashSet<String>,
}
```

Lua 游戏仍然可以自行维护普通 table；只有需要宿主绘制、碰撞、动画或输入命中的对象才注册进 `GameWorld`。

这样不会强迫所有游戏使用 ECS，又能提供统一服务。

## 缺少资源管理器

现在图片、语言、Markdown、包扫描分别拥有自己的加载逻辑，但还没有统一资源生命周期：

```
资源路径
  ↓
加载/解析
  ↓
资源句柄
  ↓
缓存与引用
  ↓
热重载
  ↓
释放
```

需要解决：

- 同一资源是否重复读取；
- 资源属于宿主还是某个游戏；
- 游戏退出时如何释放；
- 文件变化后谁失效；
- 资源加载失败是否使用占位资源；
- 一个包能否访问另一个包的资源；
- 异步加载完成后如何通知 Lua。

初版不需要复杂引用计数，可以先提供：

```
ResourceId
ResourceOwner::Host | ResourceOwner::Game(GameInstanceId)
```

以及明确的包内相对路径验证。

## 缺少终端二维空间与碰撞服务

普通引擎有物理系统，但终端游戏更需要的是轻量二维网格服务：

- 单元格坐标；
- 矩形；
- 方向；
- 网格占用；
- AABB 碰撞；
- 点与矩形命中；
- 线段或射线；
- 边界限制；
- 网格邻居查询；
- 空间分层。

例如：

```
CellPos { x, y }
CellSize { width, height }
CellRect { position, size }
GridCollider
```

这可以服务于：

- 贪吃蛇；
- 俄罗斯方块；
- 吃豆人；
- 迷宫；
- 飞机大战；
- 打砖块；
- 推箱子；
- 地牢探索。

没有必要实现刚体、质量、角动量、连续碰撞检测等普通物理引擎能力。对当前项目来说，那属于错误方向。

## 缺少游戏级渲染对象

当前底层可以画文本、边框、富文本和组件，但游戏开发者仍然需要直接理解较多绘制细节。

可以在 Canvas 之上增加较薄的游戏绘制层：

- 字符精灵（glyph sprite）；
- 文本精灵；
- 单元格矩形；
- 图块（tile）；
- 图块地图（tile map）；
- 图层与 Z 顺序；
- 相机偏移；
- 裁剪区域；
- 调色板；
- 帧动画。

例如：

```
renderer.draw_sprite({
    sprite = "player",
    x = 10,
    y = 5,
    layer = 20
})
```

底层仍然落到现有 Canvas，不需要重写渲染系统。

## 缺少动画与补间服务

Godot 的动画本质是沿时间轴修改目标属性，也能触发方法和音频。[Godot Animation](https://docs.godotengine.org/en/stable/classes/class_animation.html)

终端引擎可以提供更精简的版本：

- 延迟；
- 重复计时；
- 补间（tween）；
- 帧序列动画；
- 闪烁；
- 逐字显示；
- 颜色渐变；
- 抖动；
- 位移动画；
- 回调标记。

你已经有时间服务、定时器和对象池，因此动画服务可以建立在现有能力上，而不是另起一套时钟。

## 9. 缺少完整的音频服务

终端不负责音频，但终端游戏仍然可以播放：

- 背景音乐；
- 音效；
- 系统提示音；
- 简单蜂鸣。

这应当是可选能力：

```
无音频后端 → 静默降级
有音频后端 → 音效/音乐通道
```

初期可以只提供：

- `play`
- `stop`
- `pause`
- 音量；
- 循环；
- 音乐与音效分组。

不需要立刻实现空间音频、混响和复杂 DSP。

---

## 10. 缺少确定性、快照与回放

现有随机数和时间服务已经是基础，但还需要把它们接入游戏实例：

- 每个游戏独立随机种子；
- 保存随机数生成器状态；
- 记录语义输入；
- 固定步长推进；
- 创建游戏快照；
- 从快照恢复。

这能带来：

- 可重复测试；
- Bug 回放；
- 每日固定挑战；  
    -录像/演示模式；
- 存档恢复；
- 分数作弊检测的部分基础。

对于小游戏集合，这项能力的价值比复杂物理系统更高。

## 11. 缺少统一能力网关

未来不应让 Lua 直接拿到：

```
FileService
NetworkService
StorageService
```

而应通过包上下文调用：

```
Lua API
  ↓
GameContext
  ↓
CapabilityGuard
  ↓
路径作用域与参数校验
  ↓
实际服务
```

例如：

```
storage.write("save.json", data)
```

宿主自动限制为：

```
data/mod/game/<mod_id>/storage/save.json
```

能力网关需要同时检查：

- API 是否属于当前包类型；
- 安全模式是否允许；
- 清单是否声明；
- 用户是否授权；
- 路径是否在包作用域；
- 请求频率和数据大小是否超限。

这是 Lua 接入前必须设计好的底层边界。
