## 1. 缺少稳定的 Lua SDK

Lua 开发者需要看到的是一套一致接口，而不是 Rust 服务的直接映射。

例如：

```
local game = {}

function game:init(ctx)
end

function game:event(event)
end

function game:update(dt)
end

function game:render(renderer)
end

function game:save()
end

return game
```

同时需要提供模块：

```
tg.input
tg.render
tg.time
tg.random
tg.storage
tg.audio
tg.world
tg.ui
tg.i18n
tg.log
```

API 应当：

- 参数格式一致；
- 错误形式一致；
- 同步与异步命名清晰；
- 不泄漏 Rust 内部服务结构；
- 可以随宿主版本做兼容适配。

---

## 2. 缺少游戏开发基础库

建议提供 Lua 侧常用类型和算法：

- `Vec2`
- `Rect`
- `Grid`
- `Direction`
- `Color`
- `Timer`
- 状态机；
- 队列；
- 二维数组；
- 洗牌；
- 权重随机；
- 网格邻居；
- 简单寻路；
- 碰撞检测。

否则每个游戏都会重复实现二维坐标、网格和计时逻辑。

---

## 3. 缺少场景和游戏状态管理

开发者应当能够表达：

```
标题界面
  ↓
游戏中
  ↓
暂停
  ↓
结算
```

可以提供轻量状态栈：

```
game.states:push("pause")
game.states:replace("result")
game.states:pop()
```

它不一定对应宿主的 UI 状态机。宿主状态机管理“应用在哪个页面”，游戏状态机管理“游戏内部处于哪个阶段”，两者需要明确分开。

---

## 4. 缺少游戏内 UI

当前 Widget 更偏宿主 UI。Lua 游戏还需要受控使用：

- 标签；
- 按钮；
- 菜单；
- 对话框；
- 文本输入；
- 进度条；
- 表格；
- 滚动文本；
- 选择列表。

现有组件可以复用，但需要建立“游戏对象池所有权”，保证游戏退出后自动销毁全部组件，不能让 Lua 直接操作宿主页面的 UI 对象池。

---

## 5. 缺少开发、调试和测试工具

这是上层能否真正被使用的关键。

至少需要：

- 包创建脚手架；
- `package.json` 校验器；
- Lua API 类型提示；
- VS Code Lua 注解文件；
- 资源路径检查；
- i18n 缺失键检查；
- 无终端测试模式；
- 模拟输入；
- 固定随机种子；
- 单帧执行；
- 热重载；
- Lua 错误覆盖层；
- 调用栈与源文件位置；
- 帧耗时和内存统计；
- 示例游戏。

可以设想一组命令：

```
tg mod new snake
tg mod check ./snake
tg mod test ./snake
tg mod run ./snake
tg mod pack ./snake
```

这会比立刻制作图形化编辑器更有价值。

---

## 6. 缺少兼容性与迁移机制

包清单已经有：

```
schema_version
api.min
api.max
```

但还需要确定：

- API 废弃周期；
- 旧 API 兼容层；
- 包扫描错误格式；
- 清单迁移；
- 存档版本迁移；
- Lua API 行为是否遵循语义化版本；
- 宿主升级后旧游戏如何运行。

API 版本字段本身不能产生兼容性，真正的兼容性来自适配层和明确的行为约定。

# 哪些通用引擎能力不必实现

下面这些虽然属于 Godot、Unity 等通用引擎，但你的项目目前不需要：

- GPU 渲染管线；
- Shader；
- 材质系统；
- 3D 场景；
- 骨骼动画；
- 刚体物理；
- 光照；
- 粒子 GPU 模拟；
- 复杂空间音频；
- 导航网格；
- 可视化场景编辑器；
- 完整 ECS；
- 跨移动端和主机平台导出。

终端领域真正需要优先做的是：

```
字符网格空间
+ 稳定时序
+ 输入动作
+ 游戏实例生命周期
+ 脚本安全边界
+ 资源与对象所有权
+ 开发工具
```
