## 终端初始化不是事务性的

这是目前最值得先修的问题。

在 terminal.rs 中：

```
enable_raw_mode()?;

execute!(stdout, EnterAlternateScreen)?;
execute!(stdout, EnableMouseCapture)?;
execute!(stdout, EnableFocusChange)?;
execute!(stdout, Hide)?;
```

如果出现这种情况：

1. 原始模式成功；
2. 进入备用屏幕成功；
3. 启用鼠标或焦点事件失败；

函数会直接返回错误，但 `TerminalSurface` 尚未构造，因此它的 `Drop` 不会运行。终端可能残留在原始模式或备用屏幕。

更进一步，`TerminalService::enter()` 只是记录错误，主循环仍然继续：

```
Err(error) => {
    services.error(...);
}
```

随后 `FramePresenter` 发现没有终端 writer，就安静地返回 `Ok(())`。结果可能是：

> 程序仍在运行，但用户看不到任何界面，而且终端状态可能已经部分改变。

建议让 `enter()` 返回 `io::Result<()>`，失败时立即执行回滚，并阻止运行时继续启动。

优先级：**P0，发布前必须修复。**

## 模组国际化路径存在目录穿越风险

资源路径和入口路径已经正确验证了 `Component::Normal`，但包内国际化路径没有经过同样验证。

在 package.rs 中，国际化键被直接按 `/` 拆分：

```
let parts: Vec<&str> = key
    .split('/')
    .filter(|part| !part.is_empty())
    .collect();
```

之后直接：

```
path = path.join(dir);
```

恶意模组可以构造类似：

```
@../../../../some_file.value
```

其中的 `..` 会被当作普通字符串加入路径，从而让扫描程序尝试读取包目录之外的 JSON 文件。

目前它只能解析 `HashMap<String, String>`，利用范围有限，但这与“第三方模组运行安全”目标直接冲突。后续开放更多 API 后，安全边界会更重要。

建议国际化路径复用与 `safe_asset_path()` 相同的路径验证逻辑，明确拒绝：

- `..`
- `.`
- 根路径；
- Windows 前缀；
- 空组件；
- 非法语言代码。

优先级：**P0，开放第三方模组前必须修复。**

## 配置写入不是原子的，且失败经常被忽略

在 storage/profile.rs 中，配置使用：

```
fs::write(path, json)
```

直接覆盖文件。

如果写入过程中进程崩溃、磁盘空间不足或系统强制终止，JSON 可能变成空文件或只写入一半。下次启动解析失败后，代码会回退到默认配置。

对于包状态来说，这可能表现为：

- 所有模组重新启用；
- 调试状态丢失；
- 安全模式恢复默认；
- 用户以为设置保存成功，实际没有。

同时，UI 中多处直接忽略写入结果：

```
let _ = storage.update_game_package_state(...);
```

尽管存储服务内部会记录部分读取、解析错误，写入错误却不一定进入日志。

建议：

1. 写入同目录临时文件；
2. `flush`，必要时 `sync_all`；
3. 使用 `rename` 原子替换；
4. 更新失败时保持 UI 内存状态与磁盘状态一致；
5. 至少写入日志并向页面返回失败命令。

优先级：**P1。**

## 异步服务的成功结果会被主循环直接丢弃

async_runtime.rs 已经提供了：

- 文件读取和写入；
- 网络请求；
- 图片转换；  
    -异步等待。

但 runtime/engine_events.rs 中：

```
EngineEvent::File(_)
| EngineEvent::Image(_)
| EngineEvent::Network(_)
| EngineEvent::Time(_)
| EngineEvent::TaskFinished { .. } => {}
```

成功结果被直接丢弃，只有 `TaskFailed` 会记录日志。

这意味着服务 API 表面上可调用，但上层无法取得结果。根据 `AGENTS.md` 的“禁止为未来提前扩展”，这些服务现在处于不太理想的中间态：

- 如果当前需求不用，就不应过早形成完整公共 API；
- 如果已经要用，就必须把事件送到真正的消费者；
- 不能既暴露服务，又静默丢弃结果。

这暂时不一定导致现有功能错误，但后续 Lua API 接入时非常容易变成隐蔽故障。

优先级：**P1，在接入 Lua API 前解决。**

## 网络任务可能导致退出永久等待

网络请求使用：

```
reqwest::blocking::get(&url)
```

没有显式超时。

`AsyncRuntime::drop()` 会向工作线程发送关闭消息，然后：

```
worker.join()
```

如果某个工作线程卡在网络请求、超长休眠或不可中断的系统操作中，关闭消息只有等任务结束后才能处理，主程序退出会一直阻塞。

建议构建共享的 `reqwest::blocking::Client`，设置：

- 连接超时；
- 总请求超时；
- 响应体大小限制；
- 后续再考虑取消令牌。

优先级：**P1，开放网络 API 前处理。**

# 错误处理与日志约束

`AGENTS.md` 要求：

> 所有错误应当优先拦截并进入 log 服务。

项目目前只做到了大约七成。

做得好的地方：

- 包解析错误会转化为 `PackageEvent::Warn`；
- 配置读取和 JSON 解析会记录日志；
- 帧呈现失败会记录日志；
- 异步任务失败会统一进入引擎日志；
- panic 时会尝试恢复终端。

没有完全做到的地方：

- 输入法切换失败大量保存在 `last_error`，不进入日志；
- 终端恢复错误全部丢弃；
- 配置写入结果多处被忽略；
- 工作线程 `join()` 的 panic 结果被忽略；
- 包扫描请求发送失败被忽略；
- 日志文件自身写入失败只放在 `last_file_error`；
- `shutdown` 忽略输入法恢复失败；
- 崩溃日志写到当前目录的 `tui_crash.log`，没有遵守统一上线根目录。

并不是所有 `let _ = ...` 都应当删除。例如恢复终端时确实没有可靠的二次错误通道。但对配置写入、线程崩溃和用户操作而言，失败不能静默处理。
