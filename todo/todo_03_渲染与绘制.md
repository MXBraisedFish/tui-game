# 渲染与绘制层

## 当前阶段目标

从当前的简单行缓冲渲染升级为完整的 Canvas 像素级绘制系统，支持颜色、富文本、Unicode 宽度和增量渲染。渲染层是引擎视觉输出的核心，直接影响游戏画面质量和性能。

---

## 服务项目清单

### 3-1 RenderService 画布渲染

- **职责**：管理一个二维字符画布（Canvas），接收各方绘制指令，在帧结束时将当前画布与上一帧对比（增量 diff），仅将变化的区域输出到终端。
- **当前状态**：骨架存在。当前实现是简单的行缓冲（`Vec<String>`），仅有 `clear()`、`draw_centered()`、`present()` 三个方法，`present()` 每次都全量刷新终端。
- **待完善**：
  - 二维字符画布数据结构：替换 `Vec<String>` 为 `Vec<Vec<Cell>>`，其中 `Cell` 包含字符、前景色、背景色、样式标志。
  - 画布操作 API：`set_cell(x, y, cell)`、`fill_rect(x, y, w, h, cell)`、`draw_text(x, y, text)`、`draw_line(x1, y1, x2, y2, char)` 等。
  - Unicode 宽度计算：当前 `draw_centered` 使用 `.chars().count()` 计算字符串宽度，对 CJK 字符和 emoji 不准确。需集成 `unicode-width` crate。
  - 画布尺寸自适应：随终端 resize 自动调整画布大小。
  - 分层画布：支持多个图层叠加（背景层、游戏层、UI 层、覆盖层），最后合成输出。

### 3-2 Canvas API（Lua 可调用）

- **职责**：向 Lua 脚本暴露的绘制接口，是游戏开发者绘制游戏画面的唯一途径。需要在 RenderService 的画布之上再封装一层 Lua 友好的 API。
- **当前状态**：不存在。`LuaService` 目前仅为裸 VM，无任何 API 注册。
- **待完善**：
  - 基本绘制原语：画点、画线、画矩形（填充/边框）、画圆、写文本。
  - 裁剪区域：支持设置绘制裁剪区域，超出裁剪区的绘制操作被忽略。
  - 颜色支持：前景色和背景色设置，支持 16 色 ANSI + 256 色 + 真彩色。
  - 画笔状态：当前位置、当前颜色、当前样式等状态机的管理。
  - 双缓冲：Lua 侧绘制到后台缓冲，帧结束时由引擎 flush 到前端画布。

### 3-3 Color 颜色系统

- **职责**：定义引擎支持的颜色类型，提供颜色解析、转换和 ANSI 转义序列生成。
- **当前状态**：不存在。当前渲染无任何颜色支持，所有输出为终端默认色。
- **待完善**：
  - 颜色类型定义：16 色 ANSI 命名色、256 色调色板色、真彩色（RGB）。
  - 颜色解析：从字符串（如 `"#ff0000"`、`"red"`、`"rgb(255,0,0)"`）解析为内部颜色表示。
  - 颜色对（ColorPair）：前景色+背景色的组合，用于 Cell 的着色。
  - ANSI 转义序列生成：将颜色转换为终端可识别的转义码。

### 3-4 RichText 富文本

- **职责**：支持带混合样式的文本渲染。一段文本中可能包含多种颜色、粗体、下划线等样式变化。
- **当前状态**：不存在。
- **待完善**：
  - 富文本标记格式定义：采用何种标记语法（类 Markdown 子集、类 ANSI、或自定义标记）。
  - 解析器：将标记文本解析为带样式区段的内部表示。
  - 渲染器：将富文本渲染到 Canvas 的正确位置，处理样式切换。
  - 自动换行：富文本在指定宽度内自动换行，保持样式在换行后的连续性。
  - 文本截断：超出可用空间的文本截断并显示省略号。

### 3-5 IncrementalRenderer 增量渲染

- **职责**：比较当前帧画布与上一帧画布的差异，仅将变化的区域（脏区域）输出到终端。这是终端渲染的关键性能优化——终端的输出带宽有限，全量刷新在大画布下会导致闪烁和性能下降。
- **当前状态**：不存在。当前 `present()` 每次全量输出所有行。
- **待完善**：
  - 帧缓存：存储上一帧的画布快照。
  - Diff 算法：逐 Cell 比较当前帧与上一帧，标记变化的 Cell，合并为最小脏矩形集合。
  - 终端输出优化：将脏矩形转换为最小数量的 ANSI 光标移动+写入指令。
  - 强制全量刷新：提供 `force_repaint()` 方法，在特定场景（如首次渲染、resize 后）跳过 diff 直接全量输出。
  - 帧号跟踪：区分"首帧"（强制全量）和"后续帧"（增量）。

### 3-6 ImageCache 图片缓存

- **职责**：缓存已渲染为字符画布的图片资源。package.json 中 `display.icon` 和 `display.banner` 引用的图片在首次加载后渲染为字符画布并缓存，后续使用直接命中缓存。
- **当前状态**：不存在。
- **待完善**：
  - 图片加载：支持常见格式（PNG、JPG）加载。
  - 图片转字符画布：将像素图片转换为终端字符表示（灰度→字符映射 或 彩色→ANSI 色块）。
  - 缓存键：基于文件路径+修改时间的缓存失效策略。
  - 缓存存储：使用 CacheStore 持久化缓存到磁盘，避免每次启动重新渲染。

### 3-7 Theme 主题系统

- **职责**：管理引擎的颜色方案和布局参数，使 UI 组件和游戏画面可以使用统一的主题变量（如"主背景色"、"强调色"、"边框样式"等），并支持主题切换。
- **当前状态**：不存在。旧代码位于 `old_src/host_engine/theme/`。
- **待完善**：
  - 主题变量定义：颜色变量（bg、fg、accent、warning、error 等）+ 布局变量（边框字符、间距等）。
  - 主题文件格式：明确定义主题文件的序列化格式。
  - 内置主题：至少提供一套默认暗色主题和一套亮色主题。
  - 主题切换：运行时切换主题后，所有 UI 组件和渲染输出立即生效。
  - 主题变量解析：UI 组件通过变量名引用颜色，而非硬编码颜色值。

## 旧架构参考

### Canvas 与渲染（`old_src/host_engine/runtime/renderer/`）
旧架构的渲染器比当前骨架复杂得多：
- **Canvas**：二维 Cell 数组画布，每个 Cell 包含字符、前景色、背景色。
- **draw_* 系列方法**：提供丰富的绘制原语，包括矩形、线条、文本、富文本。
- **双缓冲机制**：Lua 绘制到后台 Canvas，帧结束时交换到前端。
- **Bridge 模式**：`Canvas::Bridge` 作为 Lua 和 Rust 之间的桥梁，将绘制命令从 Lua 传递到 Rust Canvas。

旧架构的增量渲染器（`incremental_render/`）：
- **diff**：比较两帧 Cell 数组，生成脏矩形列表。
- **frame_cache**：存储上一帧的完整画布数据。
- **terminal_output**：将脏矩形转换为 ANSI 指令序列写入终端。

### Theme（`old_src/host_engine/theme/`）
- **color_scheme**：定义了约 30 个语义颜色变量，覆盖 UI 的所有视觉元素。
- **layout_config**：定义边框样式、间距、最小尺寸等布局参数。
- **theme_manager**：加载主题文件，解析为 ColorScheme 和 LayoutConfig，供 UI 组件查询。

旧架构中 Canvas 和增量渲染是分离的模块，新架构应将其统一在 RenderService 内部，简化模块边界。

## 完成后可验证的可用项

1. 引擎启动后正确显示彩色 UI 界面。
2. 中英文混排文本对齐正确（Unicode 宽度计算准确）。
3. 连续帧之间仅输出变化的终端区域，无明显闪烁。
4. 切换主题后 UI 颜色立即变化。
5. 富文本标记被正确解析和渲染（如标题使用不同颜色）。
6. Lua 脚本调用 Canvas API 能正确在终端绘制图形。
7. 图片被加载并渲染为终端字符画布。
8. 终端 resize 后画布尺寸自动调整，画面不破碎。
