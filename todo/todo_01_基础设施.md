# 基础设施层

## 当前阶段目标

建立引擎运行的底层支撑系统，确保目录结构、语言翻译、配置读写、命令行入口和异常处理等基础能力就位。本层服务不依赖其他层，是所有上层功能的前提。

---

## 服务项目清单

### 1-1 CLI 命令行参数解析

- **职责**：解析启动参数，执行对应子命令（启动游戏、清除缓存、查看路径、显示版本等），然后按需进入交互模式或直接退出。
- **当前状态**：不存在。`main.rs` 直接调用 `host_engine::run()`，无任何参数处理。
- **待完善**：
  - 定义子命令枚举（`run` / `api` / `clear-cache` / `clear-data` / `fix` / `help` / `path` / `version`）。
  - 每个子命令的参数定义和校验。
  - 子命令执行完毕后若需启动引擎，则转交宿主编排层；否则直接退出进程。
  - `--help` 输出中英双语帮助信息。

### 1-2 Crash 与 Panic 处理

- **职责**：在发生 panic 或未捕获错误时，先恢复终端状态（退出 raw mode / alternate screen），再将崩溃信息写入日志文件，最后退出进程。这是终端应用的硬需求——如果崩溃后不恢复终端，用户屏幕将乱码且无法正常操作。
- **当前状态**：不存在。当前代码仅在正常退出路径中调用 `TerminalService::exit()`，panic 路径无任何保护。
- **待完善**：
  - 注册全局 panic hook，hook 内先恢复终端再写 crash log 再 abort。
  - Crash log 文件写入：包含时间戳、panic 消息、调用栈回溯。
  - 与 TerminalService 的配合：panic hook 需要独立于 EngineServices 也能操作终端恢复（因为 panic 时 EngineServices 可能已损坏）。

### 1-3 Logging 日志系统

- **职责**：提供分级日志输出（debug/info/warn/error），运行时写入日志文件，与 crash log 区分——crash log 仅记录崩溃，logging 记录完整运行过程。
- **当前状态**：不存在。代码中仅使用 `println!` 作临时输出。
- **待完善**：
  - 日志级别定义。
  - 日志文件路径管理和轮转策略。
  - 各服务通过日志系统报告状态，替代当前散落的 `println!` 和 TODO 注释中的 "应该写入日志"。
  - 开发模式下的终端输出 vs 发布模式下的纯文件输出。

### 1-4 Environment 环境管理

- **职责**：确定并创建引擎运行时所需的所有目录（数据目录、缓存目录、配置目录、日志目录、mods 目录、scripts 目录等），处理开发模式与发布模式的路径差异，必要时修复/初始化硬编码资产。
- **当前状态**：部分存在。`StorageService::ensure_directories()` 已创建基本目录树，包含 data/cache/profiles/logs/mods/scripts。`resolve_root_dir()` 已处理 dev/release 路径差异。
- **待完善**：
  - 从 StorageService 中剥离目录检测逻辑，划归环境管理负责。
  - 资产修复机制：硬编码的默认配置、默认键位等在首次启动或文件损坏时自动重建。
  - Bash 语言环境检测（旧架构有此功能）。
  - 环境就绪状态检查，供 Boot 序列判断是否可以继续启动。

### 1-5 Storage 存储服务

- **职责**：提供引擎数据的持久化读写，包括用户配置（Profile）和运行缓存（Cache）。向上层暴露查询接口，内部处理 JSON 序列化、文件锁、数据迁移。
- **当前状态**：骨架存在。`StorageService` 已创建目录，但 `ProfilesStore` 和 `CacheStore` 字段被注释，无实际读写功能。
- **待完善**：
  - `ProfilesStore`：存储用户偏好（语言、主题、键位映射、已启用/禁用的包列表、游戏存档状态等）。需要读、写、迁移、默认值回退。
  - `CacheStore`：存储运行时缓存（已渲染图片缓存、Lua 字节码缓存、包扫描结果缓存等）。需要读、写、失效策略。
  - 对外暴露查询方法，禁止外部直接修改内部 Map。
  - 保存操作的错误处理：写入失败不应导致引擎崩溃。

### 1-6 i18n 国际化

- **职责**：管理所有面向用户的文本翻译，支持语言切换、翻译键查找、带参数的格式化文本。所有服务的用户可见字符串最终都应通过 i18n 系统输出。
- **当前状态**：不存在。当前代码中所有字符串均为硬编码中文或英文。`PackageService` 中有多处 `// TODO: 将语言国际化` 注释标记了待翻译点。
- **待完善**：
  - 翻译键体系设计：采用层级键（如 `ui.home.title`、`error.package.not_found`）。
  - 翻译文件格式和加载机制（JSON 或 TOML，每个语言一个文件）。
  - 伪文本系统（旧架构有此概念）：在开发阶段用伪翻译检测硬编码字符串。
  - 参数插值支持（如 "第 {n} 帧"）。
  - 与 UiService、RenderService 的集成点：所有面向用户的文本渲染都应查询 i18n。
  - 支持的语言列表：至少中文简体、英文，保留扩展点。

### 1-7 Config 配置管理

- **职责**：提供用户设置的统一读写入口。与 Storage 的关系是：Storage 是底层文件读写，Config 是带语义的配置项集合（语言 = "zh-cn"，主题 = "dark"，FPS 上限 = 60 等），Config 使用 Storage 做持久化。
- **当前状态**：不存在。无统一的配置项定义。
- **待完善**：
  - 定义配置项枚举或结构体，涵盖：语言、主题、FPS 上限、内存限制、屏保空闲时间、键位方案等。
  - 配置项的默认值定义。
  - 配置变更通知机制：当某项配置被修改时，相关服务需要感知（如语言变更需通知 i18n 和 UiService 重绘）。
  - 配置校验：非法值回退到默认值。

## 旧架构参考

### CLI（`old_src/boot/cli/`）
旧架构中 CLI 是一个完整的子命令系统，定义了 Args → SubCommand → 执行 的流程。子命令包括：
- `api`：输出 API 版本信息
- `clear-cache`：清空缓存目录
- `clear-data`：清空数据目录
- `fix`：检查和修复引擎环境
- `help`：打印帮助信息
- `path`：打印引擎各目录路径
- `version`：打印版本号

CLI 解析完成后将控制权交给 Boot 序列。新架构可复用此流程设计，但需要适配新的服务容器。

### Crash / Panic（`old_src/boot/crash_log.rs`、`panic_hook.rs`）
旧架构有两个独立模块处理异常：
- `crash_log.rs`：记录崩溃日志到文件，包含调用栈和系统信息。
- `panic_hook.rs`：注册全局 panic hook，在 panic 时先恢复终端、写 crash log、打印用户友好信息后退出。
新架构必须至少实现 panic hook，否则终端状态无法保证恢复。

### i18n（`old_src/boot/i18n/`）
旧架构的 i18n 系统相当完整，包含：
- 伪文本系统：在开发阶段用带标记的伪翻译文本（如 `[HOME_TITLE]`）替代真实翻译，方便检测哪些字符串未走 i18n。
- 20+ 个按类型的翻译模块：分别处理 UI 文本、错误信息、包信息、游戏动作描述等不同类别的翻译。
- 语言键采用层级结构。
新架构可参考此设计但可简化翻译模块的数量，统一为一个翻译表即可。

### Environment（`old_src/boot/environment/`）
旧架构的环境管理负责：
- 确定数据目录、缓存目录、配置目录的绝对路径。
- 开发模式（Cargo run）与发布模式（exe 同级目录）的路径切换逻辑（新架构的 `StorageService::resolve_root_dir()` 已实现此逻辑）。
- 硬编码资产的检查和修复写入。
- Bash 语言环境的检测（Windows 下 Git Bash、WSL 等的路径探测）。

### Storage（`old_src/host_engine/storage/`）
旧架构的存储层分为：
- `profile_store.rs`：用户配置存储，使用 JSON 文件，支持读写和默认值。
- `cache_store.rs`：缓存存储，用于存储预计算数据（如图片渲染缓存），支持过期策略。
- `types.rs`：存储相关的类型定义。

旧架构中 Storage 和 Package、UI 之间存在较多的直接状态修改（通过 `&mut` 引用传递），新架构应通过请求-响应模式解耦。

## 完成后可验证的可用项

1. 引擎通过 `--help` 输出所有子命令说明。
2. 引擎通过 `--version` 输出版本号后退出，不启动终端模式。
3. 引擎通过 `--clear-cache` 清空缓存目录后退出。
4. 在代码中故意触发 panic，终端状态正常恢复且 crash log 文件生成。
5. 启动后日志文件自动创建并记录各服务初始化状态。
6. 所有目录在首次启动时自动创建。
7. Profile 文件在首次启动时写入默认值，后续启动正确读取。
8. 切换语言配置后，UI 文本随之更新。
9. 修改配置项后重启引擎，修改生效。
