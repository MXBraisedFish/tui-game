# 终端与输入层

## 当前阶段目标

完善终端模式管理和输入处理链路，建立从"物理按键"到"语义动作"的两级输入解析体系。确保终端 resize 事件被正确捕获和分发。

---

## 服务项目清单

### 2-1 TerminalService 终端管理

- **职责**：管理终端的 raw mode、交替屏幕（alternate screen）、光标显隐、终端尺寸查询。进入时启用原始模式和交替屏幕，退出时恢复原始终端状态。
- **当前状态**：基本可用。已有 `enter()` 和 `exit()` 方法，内部通过 `TerminalGuard` 的 Drop 机制保证状态恢复。
- **待完善**：
  - 终端 resize 事件的检测和通知：当用户拖动终端窗口时，需要通知渲染层和 UI 层重新计算布局。
  - 光标显隐控制：在游戏运行和屏保运行时应隐藏光标，在 UI 页面应显示光标。
  - `enter()` 失败时的错误处理：当前只有一个 TODO 注释标记了"应该写入日志"。
  - 与 Crash/Panic 层的协调：确保 panic 时终端能恢复，即使 TerminalGuard 的 Drop 未执行。

### 2-2 InputService 原始输入

- **职责**：从 crossterm 轮询原始输入事件（按键按下、释放、重复、鼠标事件），维护一个输入队列供运行时消费。
- **当前状态**：基本可用。已实现 `poll()`、`next_key()`、`consume_key()`。但仅处理键盘事件，无鼠标支持。
- **待完善**：
  - 鼠标事件支持：点击、移动、滚轮。对于 UI 页面中的列表选择和按钮点击很有必要。
  - 输入队列的容量限制：防止队列无限增长。
  - 组合键解析：将 `KeyEvent` 中的修饰键（Ctrl、Alt、Shift）与基础键组合为结构化表示。
  - 输入事件的分类：区分"UI 导航类事件"和"游戏内容类事件"，不同事件路由到不同处理链。

### 2-3 ActionService 语义动作解析

- **职责**：位于原始输入之上的一层，将原始按键映射为语义动作（如 `NavigateUp`、`Confirm`、`Quit`、`GameAction("jump")`）。这是"调用式→请求式"转换的关键环节。
- **当前状态**：不存在。当前运行时直接硬编码 `if key == KeyCode::Esc { quit }` 的判断逻辑。
- **待完善**：
  - 定义动作枚举：系统动作（退出、导航、切换页面）和游戏动作（由 package.json 的 actions 字段定义）。
  - 动作与键位的映射表，支持多键绑定到同一动作。
  - 动作的分发：根据当前上下文（在 UI 页面、在游戏中、在屏保中），同一按键可能解析为不同动作。
  - 与 KeybindingManager 的关系：ActionService 使用 KeybindingManager 提供的映射表进行解析。

### 2-4 KeybindingManager 键位管理

- **职责**：管理用户自定义的键位绑定，支持加载默认键位、用户修改、冲突检测、持久化保存。这是 ActionService 的数据来源。
- **当前状态**：不存在。旧代码位于 `old_src/host_engine/keybind/`，包含完整的键位管理系统。
- **待完善**：
  - 默认键位方案定义：至少提供一套内置默认键位（如 WASD 方向、Enter 确认、Esc 返回）。
  - 用户自定义键位：从 Profile 中读写用户修改的键位。
  - 冲突检测：当用户将两个动作绑定到同一按键时提示冲突。
  - 动作 schema：定义哪些动作是可绑定的，以及每个动作的默认按键和描述（用于在设置页面展示）。
  - 包自定义动作注册：package.json 中 `game.actions` 定义的动作为包专属动作，需要注册到全局动作表中。

### 2-5 Resize 处理

- **职责**：监听终端窗口大小变化，通知渲染层重新计算画布尺寸，通知 UI 层重新布局。这是一个横切关注点，放在终端与输入层统一管理。
- **当前状态**：不存在。当前 `RenderService` 仅在构造时获取一次终端尺寸，之后不会更新。
- **待完善**：
  - Resize 事件来源：crossterm 的 `Event::Resize`。
  - 通知机制：resize 发生后依次通知 RenderService（更新画布尺寸）、UiService（重新计算布局）、GameService（通知运行中游戏终端已变化）。
  - 最小尺寸限制：终端尺寸小于 package.json 中 `runtime.min_width` / `min_height` 时，显示提示而非尝试渲染。

## 旧架构参考

### Keybinding 系统（`old_src/host_engine/keybind/`）
旧架构的键位系统设计较为完整，包含以下模块：
- **action_schema**：定义了所有可绑定动作的元数据（动作名、默认按键、描述文本），分为系统动作和游戏动作两类。
- **binding**：单条绑定记录，将一组物理键（如 `["ctrl", "c"]`）映射到一个动作名。
- **conflict**：冲突检测逻辑，检查同一按键是否被多个动作绑定。
- **default**：默认键位方案的硬编码定义。
- **manager**：键位管理器，整合以上所有模块，对外提供 `resolve(key_event) -> Option<Action>` 的查询接口。
- **resolver**：上下文感知的解析器，根据当前是"UI 模式"还是"游戏模式"返回不同的动作。

这个分层设计值得新架构直接沿袭：物理输入→绑定查询→上下文解析→语义动作。

新架构的改进方向：旧架构中 KeybindingManager 直接访问 ProfileStore 做持久化，存在跨服务直接修改。新架构应改为 ActionService 向 KeybindingManager 查询映射表，KeybindingManager 通过请求方式从 StorageService 获取持久化的键位数据。

## 完成后可验证的可用项

1. 启动引擎后终端进入 raw mode + 交替屏幕，按 Esc 退出后终端恢复正常。
2. 拖动终端窗口改变大小，RenderService 的画布尺寸随之更新。
3. 按默认键位（如方向键）在 UI 页面间导航正常。
4. 在设置页面修改键位后，新键位立即生效。
5. 同一按键绑定两个动作时，引擎提示冲突并拒绝保存。
6. 鼠标点击 UI 列表项可选中。
7. 游戏运行中按键被路由到游戏而非 UI 导航。
